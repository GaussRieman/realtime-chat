# Audio Anything 暂停与结束会话设计规格

日期：2026-07-16
状态：用户已确认
范围：实时语音会话生命周期

## 1. 目标

明确区分“暂停”“结束”和“断线”：

- 暂停是用户主动临时挂起同一次会话，保留 WebSocket、模型上下文和字幕。
- 结束是用户主动关闭会话，完成字幕快照与后台分析，然后回到待开始状态。
- `CONNECTION PAUSED` 只用于真实网络或上游异常，不得在主动结束后出现。

## 2. 暂停与继续

### 2.1 暂停

暂停按钮只在已建立连接的会话中可用，不在连接中或断线状态显示为可操作。

用户点击暂停后：

1. 状态切换为 `paused`，标题显示“会话已暂停”。
2. 立即停止麦克风上传并禁用麦克风轨道。
3. 立即停止并清空本地 AI 播放队列。
4. 若存在活跃 AI 响应，则使当前响应代次失效并向上游发送 `response.cancel`。
5. 若 `input_audio_buffer.speech_started` 已发生但尚未 `speech_stopped`，先发送已确认支持的 `input_audio_buffer.commit`，把暂停前音频明确结束为一条可能不完整的用户回合；这条用户字幕可以保留，但模型回复必须被抑制。
6. 设置“抑制响应直到恢复后下一次 `speech_started`”门闩。门闩期间任何迟到的 `response.created` 都立即进入失效代次并再次发送 `response.cancel`；其音频和流式字幕全部丢弃。
7. 如果暂停前已经存在流式 AI 字幕，则把已显示文本标记为 `interrupted`，不得留下永久 `streaming` 条目。
8. 停止会话计时，不关闭 WebSocket，不轮换 `conversationId`。
9. 保留字幕、当前音色、用户原有静音选择和模型上下文。
10. 暂停期间到达的音频 delta、VAD 状态和响应状态不得改变 `paused`；已经在途的用户最终字幕可以继续合并。
11. 暂停不生成会话分析。

暂停按钮变为“继续”。暂停期间仍可点击“结束”；静音控件禁用，避免与暂停状态产生歧义。

### 2.2 继续

用户点击继续后：

1. 恢复同一个 WebSocket 和 `conversationId`。
2. 根据暂停前的用户静音选择恢复麦克风轨道；如果原本静音，则继续保持静音。
3. 重新开始累计活跃会话时长，暂停时间不计入报告时长。
4. 状态回到 `listening`，但响应抑制门闩继续保持，直到服务端检测到恢复后的下一次 `input_audio_buffer.speech_started`。该事件先使暂停前的迟到响应失效，再解除门闩；随后新用户回合的响应可以正常播放。
5. 用户可以直接继续说话；暂停前未完成的用户语音不会与恢复后的语音拼接成同一回合。

如果暂停期间真实链路断开，则进入 `disconnected`，继续按钮不可用，只提供重新连接。

## 3. 主动结束

用户从任意已连接状态（包括 `paused`）点击结束后：

1. 短暂进入 `ending`，立即停止麦克风和本地播放。
2. 最多保留 1 秒字幕信令收尾，之后关闭 WebSocket。
3. WebSocket 客户端将这次关闭标记为 `intentional`；对应 `close` 事件不能派发断线事件。
4. 生成不可变字幕快照，触发 Qwen 3.7 Max 后台分析。
5. 最终 UI 状态切换为 `idle`，显示“开始实时对话”，字幕和分析入口继续保留。
6. 用户再次开始时创建新的 `conversationId` 并清空实时字幕；上一会话分析仍按既有规则隔离。

主动关闭过程中即使底层 `close` 事件晚到，也不能覆盖 `idle` 或显示 `CONNECTION PAUSED`。

## 4. 控件与状态

```text
connecting    控件禁用，等待链路就绪
listening     静音 / 暂停 / 结束 可用
generating    静音 / 暂停 / 结束 可用
speaking      静音 / 暂停 / 结束 可用
interrupted   静音 / 暂停 / 结束 可用
paused        继续 / 结束 可用，静音禁用
ending        全部会话控件禁用
idle          显示开始入口
disconnected  显示重新连接入口
```

暂停按钮使用暂停/播放图标和可见文字或可访问名称，触控区域不小于 44 × 44 CSS 像素。暂停波形静止，不使用错误红色。

## 5. 实现边界

- `RealtimeClient` 负责区分主动关闭和非主动关闭；主动关闭不派发 `disconnect`。
- `useRealtimeConversation` 负责 `paused` 状态、计时分段、麦克风/播放控制和字幕门控。
- 实时协议允许列表增加文档已确认的 `input_audio_buffer.commit`，只在暂停时终止已开始但未结束的用户回合。
- 代理在暂停的无音频期间维持传输层 WebSocket 心跳；心跳不得进入模型对话内容。
- `App` 只根据 hook 暴露的状态与操作渲染暂停/继续按钮。
- 会话分析 hook 不感知暂停，只在主动结束返回快照后运行。

不得通过延时设置 `endingRef` 或忽略所有 `close` 事件来规避竞态，因为这会掩盖真实断线。

## 6. 错误与竞态

- 暂停与结束重复点击必须幂等。
- 暂停期间迟到音频不得播放。
- 暂停前的用户回合通过 `input_audio_buffer.commit` 明确结束；恢复后的语音必须形成新回合。
- 暂停或刚恢复时迟到的 `response.created` 必须取消，直到新的 `speech_started` 解除响应抑制门闩。
- 被暂停打断的 AI 流式字幕必须转为 `interrupted`。
- 主动结束的迟到 `close` 不得进入断线状态。
- 网络断线仍必须派发 `disconnect` 并保留字幕。
- 上游取消返回“Conversation has no active response”继续按既有规则忽略。
- 结束清理异常时仍应尽力返回字幕快照并回到待开始状态，不能永久卡在 `ending`。

## 7. 验收

- 正常会话点击暂停后，输入、输出和计时立即停止，字幕与连接保留。
- 点击继续后使用同一 `conversationId` 恢复聆听，暂停时间不计时。
- 暂停前静音时，继续后仍保持静音。
- 用户说话中暂停时，暂停前内容可以形成一条不完整字幕，但恢复后的语音不会与其拼接。
- 从 generating 或 speaking 暂停后，迟到响应不会在暂停期间或刚恢复时播放。
- 长时间暂停时 WebSocket 心跳保持连接；暂停期间断线仍进入重连。
- 暂停与结束并发点击保持幂等，不会重复关闭或重复分析。
- 暂停期间点击结束，最终进入待开始并触发一次分析。
- listening、speaking 和 paused 状态主动结束后都不显示 `CONNECTION PAUSED`。
- 真实 WebSocket 异常关闭仍显示 `CONNECTION PAUSED`。
- 全量测试和生产构建通过。
