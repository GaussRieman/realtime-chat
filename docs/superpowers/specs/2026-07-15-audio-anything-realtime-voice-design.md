# Audio Anything 实时语音 WebAPP 设计规格

日期：2026-07-15  
状态：已批准并进入实施  
目标模型：`qwen-audio-3.0-realtime-plus`  
服务区域：阿里云百炼，中国内地，华北 2（北京）

## 1. 产品定义

Audio Anything 是一个面向单用户的浏览器实时语音对话应用。用户点击一次“开始实时对话”，授权麦克风后即可持续与千问交流，不需要按住按钮。模型通过服务端 VAD 自动判断说话和停顿；用户可在 AI 播放回答时直接插话，应用应立即停止当前回答并切回聆听。

首版的核心价值不是传统聊天记录，而是让“正在连接、聆听、思考、说话和被打断”都具有清晰、低延迟的声音反馈。

### 1.1 成功标准

- 用户只需一次点击即可进入持续全双工会话。
- API Key 永远不进入浏览器代码、响应或日志。
- 双方语音都有逐句实时字幕，并能区分用户、AI 和系统事件。
- 用户插话后，AI 的本地播放队列立即清空，不继续播放旧回答。
- 断线时保留当前页面中的对话记录，并提供单一明确的重连操作。
- 页面在桌面端和移动端均可完整使用，不出现横向滚动或关键控件遮挡。

### 1.2 首版范围

- 点击一次开始或结束持续语音会话。
- 麦克风采集、16 kHz PCM 输入和 24 kHz PCM 输出。
- 服务端 VAD 自动断句。
- AI 回答期间的用户插话打断。
- 用户与 AI 的实时字幕。
- 连接状态、会话计时、输入输出采样率和延迟展示。
- 麦克风静音、字幕开关、音色切换。
- 当前设备上的临时对话记录。
- 麦克风权限、网络断线和服务端错误反馈。

### 1.3 非目标

- 用户登录、账户系统或多租户管理。
- 云端会话历史、跨设备同步或数据库。
- 多用户房间、会议或并发通话管理后台。
- 文件上传、屏幕共享、图像输入或视频输入。
- Function Calling、联网搜索和复杂工具编排。
- 计费面板、Token 统计或配额管理。

## 2. 已确认的产品决策

- 对话方式：点击一次开始，之后持续全双工对话。
- 视觉方向：声波实验室中的“信号指挥台”。
- 核心布局：左侧实时声场与控制，右侧双向字幕流。
- 技术架构：React + Vite 前端，Node.js + WebSocket 安全代理。
- 音色默认值：`longanqian`。
- 系统音色：`longanqian`、`longanlingxin`、`longanlingxi`、`longanxiaoxin`、`longanlufeng`。
- 不允许浏览器直连百炼，避免暴露 `DASHSCOPE_API_KEY`。

## 3. 体验流程

### 3.1 首次进入

1. 页面显示信号指挥台的空闲态，背景信号线保持静止。
2. 中央只保留一个高优先级操作：“开始实时对话”。
3. 辅助说明明确：点击后需要麦克风权限，之后无需按住按钮。
4. 用户点击开始后，浏览器请求麦克风权限并初始化音频上下文。
5. 权限通过后建立到本地代理的 WebSocket；连接完成才进入持续收音。

若用户拒绝权限，页面不反复弹窗，而是说明如何在浏览器设置中恢复权限，并提供“重新检查权限”。

### 3.2 持续对话

1. 用户说话时，主波形使用用户信号色，标题显示“正在听你说”。
2. 服务端判断用户停顿后，状态切换为“正在生成回应”。
3. 收到 AI 音频后，主波形切换为 AI 信号色，标题显示“AI 正在回应”。
4. 用户与 AI 的转写分别进入右侧字幕流；进行中的句子带实时标记，完成后固定。
5. 会话期间麦克风持续上传，除非用户手动静音或结束会话。

### 3.3 插话打断

1. AI 回答期间检测到用户开始说话。
2. 本地输出调度器立即停止已调度的播放节点并清空待播 PCM 队列。
3. 当前响应代次立即失效；之后到达的旧响应音频和字幕 delta 必须被丢弃，不得重新进入播放队列。
4. 若真实协议支持响应取消，代理同时向上游发送取消事件，但本地正确性不依赖取消事件及时生效。
5. 主波形切换为用户信号色，标题显示“已停止回答，正在听你说”。
6. AI 字幕保留已完成部分，并在截断位置显示省略符号。
7. 系统字幕记录“AI 回答已被你打断”。
8. 新的用户音频继续进入同一会话，不新建连接。

打断不能依赖用户点击按钮。服务端语音开始事件是主要信号；客户端的输入能量检测可用于更快地停止本地播放，但不负责提交回合或替代服务端 VAD。

### 3.4 结束和断线

- 用户主动结束：停止麦克风轨道、清空播放队列、关闭 WebSocket，将当前字幕保留在页面中。
- 短暂断线：立即停止上传和播放，保留字幕，显示带“重新连接”按钮的异常层。
- 重连：创建新模型会话。首版不自动把旧上下文重新发送给模型；界面应提示“页面记录已保留，新连接将开始新的模型上下文”。
- 服务端错误：显示服务端提供的可读错误；UI 不展示堆栈、Authorization 或原始密钥信息。

## 4. 视觉与界面规格

### 4.1 视觉语言

“信号指挥台”采用深色沉浸舞台、青色实时信号、暖色用户输入和细密网格。视觉重点是横向波形、低延迟指标和逐句字幕，而不是常规气泡聊天。

- 背景：近黑蓝，叠加低对比网格和中心径向光晕。
- AI 信号：荧光青，用于 AI 波形、实时字幕边线和活动状态。
- 用户信号：暖琥珀，用于用户波形和用户字幕边线。
- 成功状态：偏青绿，仅用于连接正常提示。
- 错误状态：克制的红色，仅用于断线和不可恢复错误。
- 文字：高对比冷白；辅助文字至少达到 WCAG AA 的可读对比度。
- 字体：标题使用具有几何感但可读的展示字体，正文使用清晰的中文无衬线字体；不得使用默认 Arial/Inter 作为产品视觉主字体。

持续动画只用于表达真实音频活动。静止、断线和减少动态效果模式下，波形不循环装饰动画。

### 4.2 桌面布局

- 顶栏：品牌、产品副标题、连接与延迟状态、设置入口。
- 主区左侧：会话编号、模型标识、状态文案、主波形和会话控制。
- 主区右侧：会话计时和双向字幕流。
- 控制区：静音、结束、字幕开关和主页面原生音色下拉框，不使用设置弹窗。
- 对话中可以预选下一次会话的音色；当前会话继续显示并使用建连时的音色。
- 结束按钮始终可见，颜色与其他控制明显区分，但不通过放大造成布局跳动。

### 4.3 移动布局

- 顶栏隐藏非必要副标题，只保留品牌、连接点和设置。
- 主声场位于首屏上部，控制区在声场下方自动换行。
- 字幕流移动到主声场下方，不使用横向滚动。
- 结束、静音、字幕和音色控制保持至少 44 × 44 CSS 像素的触控区域。
- 适配 320 px 起的视口宽度，并在 375 px、768 px、1024 px 和 1440 px 验证。

### 4.4 四个关键状态

#### 开始前

- 背景指挥台可见但弱化，主波形静止。
- 中央显示麦克风入口和唯一主按钮。
- 明确说明持续对话方式与麦克风用途。

#### 对话中

- 波形根据实际输入或输出电平变化。
- AI 正在说话时显示实时链路延迟和流式字幕。
- 状态不只通过颜色表达，同时必须有文字标签。

#### 用户插话

- 本地 AI 音频立即停止。
- 波形从 AI 色切到用户色，并显示明显的纵向切换标记。
- 系统字幕记录打断事实，避免用户误认为音频故障。

#### 连接异常

- 音频舞台静止并由半透明异常层覆盖。
- 说明字幕记录仍保留，但模型上下文将在新连接中重置。
- 只提供“重新连接”一个主要恢复动作。

## 5. 系统架构

```text
Browser / React
  ├─ Microphone + AudioWorklet
  ├─ 16 kHz PCM encoder
  ├─ Realtime state machine
  ├─ Transcript store
  └─ 24 kHz PCM playback scheduler
             │ local WebSocket
             ▼
Node.js realtime proxy
  ├─ Origin and payload validation
  ├─ API key injection
  ├─ Session lifecycle
  └─ Event pass-through / sanitization
             │ authorized WebSocket
             ▼
Alibaba Cloud Bailian
qwen-audio-3.0-realtime-plus
```

前后端保存在同一项目中，但以清晰边界组织：前端不知道云端密钥，代理不处理 UI 状态，音频模块不直接修改 React 组件。

### 5.1 前端模块

- `RealtimeClient`：维护浏览器到代理的 WebSocket，发送客户端事件并分发服务端事件。
- `MicrophoneCapture`：请求权限、使用会话共享的 AudioContext 读取麦克风、向工作线程提供原始浮点采样。
- `Pcm16Resampler`：将浏览器常见的 44.1/48 kHz 输入转换为单声道 16 kHz、16-bit little-endian PCM。
- `AudioPlaybackQueue`：使用同一个会话 AudioContext，将返回的 24 kHz PCM 解码为浮点采样，连续调度播放，并支持原子清空。
- `ResponseGate`：按上游 response ID 管理响应代次；打断后使当前代次失效，拒绝所有迟到的旧音频和字幕事件。
- `ConversationMachine`：维护空闲、连接中、聆听、生成、说话、被打断、断线和结束状态。
- `TranscriptStore`：按 item ID 合并流式字幕，区分进行中、完成和被截断状态。
- `VoiceConsole`：只消费状态和操作接口，负责信号指挥台界面。

### 5.2 代理模块

- `UpgradeHandler`：只接受允许来源的 WebSocket 升级请求。
- `BailianSession`：使用服务端环境变量建立百炼连接，并在连接后发送 `session.update`。
- `EventRelay`：转发允许的实时事件，限制单条消息大小和连接速率。
- `ErrorSanitizer`：将上游错误转换为前端可读格式，移除请求头和敏感字段。
- `HealthRoute`：提供无敏感信息的健康检查，便于本地启动和部署验证。

## 6. 实时协议与音频

### 6.1 上游连接

代理连接：

`wss://llm-vdcmboq80tbpljv5.cn-beijing.maas.aliyuncs.com/api-ws/v1/realtime?model=qwen-audio-3.0-realtime-plus`

请求头：

`Authorization: Bearer ${DASHSCOPE_API_KEY}`

密钥只从代理进程环境读取。缺失时服务启动失败，并输出不包含密钥值的配置错误。

### 6.2 会话初始化

连接成功后代理发送：

```json
{
  "type": "session.update",
  "session": {
    "modalities": ["text", "audio"],
    "voice": "longanqian",
    "turn_detection": {
      "type": "server_vad",
      "threshold": 0.5,
      "silence_duration_ms": 500
    }
  }
}
```

音色和 VAD 参数由服务端允许列表生成，不接受前端传入任意 JSON 覆盖整个 session。

上游连接建立后，代理等待 `session.created` 再发送第一次 `session.update`；只有收到 `session.updated` 后才向浏览器发送 `proxy.ready`。浏览器在 ready 前不会上传音频，避免首次连接时会话配置尚未生效。

### 6.3 输入音频

- 采集：单声道 Float32，采样率由浏览器设备决定。
- 转换：重采样为 16 kHz，并夹紧转换为 signed PCM16 little-endian。
- 分块：以约 100 ms 为一块，减少消息开销同时控制交互延迟。
- 传输：按控制台示例，Base64 编码后发送 `input_audio_buffer.append`；该事件名和载荷结构仍须在协议探测中确认后固化到适配层。
- 静音：停止发送新块，但保持连接；恢复后继续同一会话。

### 6.4 输出音频

- 处理 `response.audio.delta`，Base64 解码为 24 kHz PCM16 little-endian。
- 使用独立播放时钟连续调度，避免每块音频之间出现爆音或空隙。
- 队列设置短缓冲上限，防止网络抖动后积累过长的过时回答。
- 每个音频块必须绑定上游 response ID；如果上游事件没有稳定 ID，则由事件适配层在响应开始时生成单调递增的本地代次。
- 收到打断信号时，使当前 response ID 或本地代次失效，停止全部已调度节点，清空未播放数据并重置调度时钟。
- 失效代次的任何迟到 `response.audio.delta` 都直接丢弃；若上游支持取消事件，再同时发送取消以节省 Token 和带宽。

### 6.5 字幕事件

- `conversation.item.input_audio_transcription.completed`：完成用户字幕。
- `response.audio_transcript.done`：完成 AI 字幕。
- 若上游提供字幕 delta 事件，则在同一 item 上增量更新；若没有，则显示“正在识别/正在说话”，完成事件到达后一次写入。
- `error`：经过代理清理后进入系统字幕，并触发可恢复或不可恢复状态。

实现前应使用有效账号做一次协议探测，记录该模型实际返回的完整事件类型，尤其是语音开始、响应取消和字幕 delta 事件。未在已提供示例中确认的事件不得硬编码为唯一控制路径。

已提供控制台示例可确认 `session.update`、`input_audio_buffer.append`、`response.audio.delta`、`conversation.item.input_audio_transcription.completed`、`response.audio_transcript.done` 和 `error` 的基本方向，但仍须探测真实事件载荷、ID 字段和时序。所有上游差异只进入协议适配层。

## 7. 状态模型

```text
idle
  -> connecting
  -> listening
  -> generating
  -> speaking
  -> interrupted -> listening
  -> disconnected -> connecting
  -> ended
```

- 权限拒绝从 `connecting` 返回 `idle`，并附带可操作错误。
- 所有活跃状态都可通过用户点击结束进入 `ended`。
- 只有存在打开的 WebSocket 和有效麦克风轨道时，UI 才能显示“正在聆听”。
- `speaking` 仅由收到并调度 AI 音频触发，不以文本响应到达代替。
- React 状态机与实际资源生命周期必须一致；关闭状态必须释放 MediaStreamTrack、AudioWorklet、AudioContext 调度节点和 WebSocket 监听器。

## 8. 本地数据

首版不使用数据库。当前会话字幕保存在内存中，并镜像到 `sessionStorage`，用于页面误刷新后的恢复提示。用户主动结束后记录保留到当前浏览器标签页关闭；新建会话时覆盖上一份临时记录。

每条字幕包含：

- `id`
- `role`: `user | assistant | system`
- `text`
- `status`: `streaming | completed | interrupted | error`
- `startedAt`
- `completedAt`

不得把原始音频、API Key、Authorization 请求头或完整上游错误对象写入浏览器存储。

## 9. 安全与隐私

- `.env`、`.env.*` 和真实密钥必须加入 Git 忽略规则；只提交 `.env.example`。
- 代理只允许受信任的站点 Origin，生产环境不得使用通配符。
- WebSocket 消息需验证 JSON 结构、事件类型和大小；拒绝未知客户端控制事件。
- 每个浏览器连接只创建一个上游会话，并设置连接数、消息速率和空闲超时限制。
- 日志仅记录连接 ID、事件类型、耗时和错误码，不记录音频 Base64、完整字幕或请求头。
- 页面在开始前解释麦克风用途；结束会话后立即停止设备轨道。
- 生产环境必须使用 HTTPS/WSS，否则大多数浏览器不会开放麦克风。

## 10. 错误处理

- 不支持麦克风：提示当前浏览器不支持实时语音，并禁用开始按钮。
- 权限拒绝：提供浏览器设置恢复指引和重新检查按钮。
- 代理不可达：显示“服务暂不可用”，不进入伪连接状态。
- API Key 缺失或无效：代理返回稳定错误码，前端提示服务配置问题。
- 上游限流：显示稍后重试，不自动高频重连。
- `invalid_request_error`：记录为系统提示但保持连接；“Conversation has no active response”属于取消竞态，直接忽略。
- `server_error`：停止音频资源并进入重连状态。
- 网络断线：停止播放与上传，保留字幕，允许用户主动重连。
- 音频解码异常：跳过损坏块并记录匿名错误；连续失败时结束会话。
- 页面隐藏或设备切换：不中断连接，但根据浏览器能力重新校验音频上下文；恢复失败时提示用户重新连接。

自动重连最多用于首次连接前的瞬时失败，并采用有限退避。活跃会话断线后由用户确认重连，避免在用户不知情时再次打开麦克风。

## 11. 可访问性

- 所有图标按钮都提供可读标签，状态不只依赖颜色。
- 键盘可完成开始、静音、结束、字幕和音色操作。
- 焦点样式保持明显，主控制触控区域至少 44 × 44 CSS 像素。
- 状态变化通过节制的 `aria-live` 文本播报；音频波形本身对屏幕阅读器只提供摘要。
- 遵循 `prefers-reduced-motion`，关闭非必要过渡和循环波形动画。
- 字幕开关只控制视觉显示，不停止生成字幕记录，重新开启后可恢复当前会话字幕。

## 12. 测试策略

### 12.1 单元测试

- Float32 到 PCM16 的边界值、钳位和字节序。
- 48 kHz 与 44.1 kHz 到 16 kHz 的重采样长度和频率误差。
- 24 kHz PCM 解码和连续播放时间计算。
- 打断时播放队列的同步清空和重复调用幂等性。
- 失效 response ID 或响应代次的迟到音频、字幕事件不会进入播放与字幕存储。
- 状态机合法转换和资源清理。
- 流式字幕合并、完成和截断。
- 代理的事件允许列表、大小限制和错误清理。

## 13. 验收标准

- 从点击开始到连接状态可见，不出现无反馈等待。
- 正常网络下，用户停顿后能感知到流式响应；UI 不人为等待完整回答。
- AI 播放期间插话，本地旧音频在可感知上立即停止，且不在稍后恢复。
- 用户与 AI 字幕身份、时间和进行中状态清晰。
- 结束后浏览器麦克风占用标识消失。
- 断线、权限拒绝和无效密钥都有清晰且不泄密的提示。
- 桌面和移动布局均无横向滚动，主操作可触达。
- 生产构建中搜索不到 `DASHSCOPE_API_KEY` 或真实密钥值。

界面显示的“链路延迟”定义为浏览器发送代理 ping 到收到代理 pong 的 WebSocket 往返时延，取最近 5 次成功采样的移动中位数。它用于反映浏览器到代理的连接质量，不宣称是模型端到端响应时延；端到端体验通过用户停顿到首个 AI 音频块的时间单独记录在开发诊断中，不作为首版常驻 UI 指标。

## 14. 实施边界与待验证项

以下项目不改变已批准设计，但必须在实现阶段用真实接口确认：

- 控制台示例所示 `input_audio_buffer.append` 的真实载荷限制，以及该模型实际发送的响应开始、语音开始、语音结束、响应取消和字幕 delta 事件名称与 ID 字段。
- 服务端 VAD 是否自动取消正在生成的回答；若不自动取消，确认模型支持的取消事件。
- 浏览器端到代理的最佳音频块大小，初始值采用约 100 ms，再以实际延迟和吞吐调整。
- 首版默认 `longanqian`，UI 展示文档确认的 5 个系统音色。

音色仅在第一次 `session.update` 中生效。主页面音色下拉始终可操作：未连接时选择用于本次会话，活跃会话中更改则保存为下一次会话配置；当前会话的实际音色不变。

如果上游协议与示例存在差异，应只调整 `BailianSession` 和事件适配层，不改变 UI 状态机与音频模块的公共接口。
